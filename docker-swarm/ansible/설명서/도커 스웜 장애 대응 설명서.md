# 🛠 Docker Swarm 매니저 노드 장애 복구 (Recovery Playbook)

> Ansible 기반으로 Docker Swarm 매니저 노드가 1대 / 2대 / 3대(전체) 다운되었을 때
> 상황별로 자동 복구를 수행하는 role 모음입니다.
> Prometheus 알림 → Alertmanager → webhook(`ansible-webhook.py`)을 거쳐
> 자동으로 트리거되며, **어떤 노드가 다운됐는지는 알림 라벨에서 동적으로 추출**됩니다.

## 소개

3대의 매니저(`swarm-mg1`, `swarm-mg2`, `swarm-mg3`)로 구성된 Docker Swarm 클러스터에서,
노드가 다운되었을 때 몇 대가 죽었는지에 따라 복구 절차와 위험도가 완전히 달라집니다.

- 1대만 죽으면 → 나머지 2대로 쿼럼(quorum)이 유지되므로 **단순 재조인**만으로 복구됩니다.
- 2대가 죽으면 → 쿼럼이 깨지므로, 생존한 1대가 **자신의 raft 상태로 강제 재구성**해야 합니다.
- 3대 전부 죽으면 → 살아있는 raft 상태 자체가 없으므로, **S3에 백업해둔 raft 데이터를 복원**한 뒤 재구성해야 합니다.

이 세 가지를 각각 `swarm/recovery1`, `swarm/recovery2`, `swarm/recovery3` role로 분리해 두었으며,
그 앞에 **다운된 인스턴스를 OpenStack에서 실제로 삭제·재생성하는 `swarm/recovery-provision`
단계**가 공통으로 선행됩니다.

<br>

## 복구 시나리오 요약

| 시나리오 | 다운된 대수 | 쿼럼 상태 | `down_nodes` 항목 수 | 핵심 동작 |
| --- | :---: | --- | :---: | --- |
| **Recovery1** | 1 / 3 | 유지됨 (2대 생존) | 1개 | 생존 리더(`find-leader`가 찾음)의 join 토큰으로 재생성 노드를 재조인 |
| **Recovery2** | 2 / 3 | 상실됨 (1대 생존) | 정확히 2개 (검증 실패 시 즉시 중단) | 생존 노드가 `--force-new-cluster`로 자신의 raft 기반 재구성 후, 2대 재조인 |
| **Recovery3** | 3 / 3 (전멸) | 없음 (라이브 상태 없음) | (없음, S3 최신 백업 자동 탐색) | S3 최신 백업 다운로드 → 대상 노드에 swarm 디렉터리 복원 → `--force-new-cluster` → 나머지 2대 재조인 |

<br>

## 전체 실행 흐름

```yaml
- name: Provision OpenStack Instances & Handle Load Balancer Pool
  hosts: localhost
  tags: [swarm_recovery1, swarm_recovery2, swarm_recovery3]
  roles: [swarm/recovery-provision]

- name: Install Docker
  hosts: swarm_managers
  tags: [docker, swarm_setup, swarm_recovery3, swarm_recovery2, swarm_recovery1]
  roles: [swarm/docker]

- name: Discover current swarm leader
  hosts: swarm_managers
  tags: [swarm_recovery1]
  roles: [swarm/find-leader]

- name: Recovery swarm when 1 node down
  tags: [swarm_recovery1]
  roles: [swarm/recovery1]

- name: Recovery swarm when 2 nodes down
  tags: [swarm_recovery2]
  roles: [swarm/recovery2]

- name: Recovery swarm when 3 nodes down
  tags: [swarm_recovery3]
  roles: [swarm/recovery3]
```

> ⚠️ **Recovery1만 `find-leader`(swarm_leader) 태그를 함께 거칩니다.**
> Recovery2/3는 쿼럼 자체가 깨진 상황을 가정하므로, `docker node ls`로 리더를 조회하는
> `find-leader` 방식이 통하지 않습니다. 대신 role 내부에서 "다운되지 않은 노드"를
> 직접 생존 노드로 지정해 처리합니다.

> ⚠️ **`recovery-provision`은 세 태그(`swarm_recovery1/2/3`) 모두에서 공통으로 먼저 실행됩니다.**
> 노드가 "이미 재생성되어 있다"는 걸 전제하지 않고, 이 단계 자체가 OpenStack 인스턴스를
> 직접 삭제·재생성합니다. 즉 사람이 미리 인스턴스를 지우고 다시 만들어둘 필요가 없습니다.

<br>

## 사전 조건

- Recovery3를 쓰려면 사전에 `swarm_backup` 태그로 S3에 백업이 최소 1회 이상 생성되어 있어야 합니다.
  백업이 하나도 없으면 `sort_by(Contents,&LastModified)` 조회 단계에서 즉시 실패합니다.
- S3 접근을 위한 AWS 프로필과 버킷명이 vault 등에 정의되어 있어야 합니다.
- `recovery-provision` 단계가 OpenStack Load Balancer pool 조작, 인스턴스 삭제/생성, 네트워크
  포트 관리까지 수행하므로, 실행 계정에 해당 OpenStack 프로젝트에 대한 권한이 있어야 합니다.

<br>

## 입력 변수: `down_nodes`

세 recovery role 모두 **하나의 공통 변수 `down_nodes`**를 받습니다. (기존 문서의
`down_node_hostname` / `down_node_hostnames`라는 이름은 실제 구현과 다릅니다.)

```json
{
  "down_nodes": [
    {"name": "swarm-mg2", "fixed_ip": "192.168.1.21"},
    {"name": "swarm-mg3", "fixed_ip": "192.168.1.22"}
  ]
}
```

| 필드 | 설명 |
| --- | --- |
| `name` | 다운된 노드의 인벤토리/OpenStack 인스턴스 이름 (예: `swarm-mg2`) |
| `fixed_ip` | 그 노드가 쓰던 고정 IP. 재생성 시 동일한 IP로 포트를 다시 만들어 붙이는 데 사용됨 |

- Recovery1은 리스트에 **정확히 1개**가 들어와야 합니다.
- Recovery2는 리스트에 **정확히 2개**가 들어와야 하며, 아니면
  `"recovery2는 2대의 장애 노드가 리스트(down_nodes) 형태로 전달되어야 하며..."`
  라는 assert 에러로 즉시 실패합니다.
- Recovery3는 이 변수를 쓰지 않고 S3에서 자동으로 대상을 판별합니다.

<br>

## 시나리오별 상세

### 사전 단계: 노드 재프로비저닝 (recovery-provision)

세 시나리오 모두 실제 복구(재조인) 전에 이 단계를 거칩니다.

**동작 순서** (`down_nodes`의 각 항목에 대해 반복)

1. 웹훅으로부터 받은 입력값(`down_nodes`)을 검증합니다.
2. 해당 노드를 OpenStack Load Balancer pool에서 제거합니다 (`state=absent`).
3. 죽은 인스턴스를 OpenStack에서 완전히 삭제합니다.
4. 같은 이름·같은 고정 IP로 네트워크 포트를 다시 확보합니다
   (`Ensure network port exists with identical Fixed IP`).
5. 그 포트를 사용해 인스턴스를 재생성합니다.
6. 새로 태어난 인스턴스를 동적 인벤토리 그룹에 등록합니다.
7. 다시 Load Balancer pool에 멤버로 추가합니다.
8. SSH 연결이 응답할 때까지 대기합니다.
9. 이후 공통 플레이(Node Exporter 설치, Docker 설치)가 이어서 실행됩니다.

> 인스턴스 삭제 후 남는 **네트워크 포트가 자동으로 정리되지 않는 경우**가 있어,
> 같은 고정 IP로 재생성을 시도할 때 `Fixed IP address ... already in use`
> 에러가 날 수 있습니다. 이 경우 `openstack port list`로 유령 포트를 찾아
> 수동으로 `openstack port delete`해야 합니다.

<br>

### Recovery1: 1대 다운 (쿼럼 유지)

**전제**: 3대 중 1대만 다운됨. 나머지 2대는 정상 동작 중이라 쿼럼이 살아있습니다.

**필수 변수**

| 변수 | 필수 | 설명 |
| --- | :---: | --- |
| `down_nodes` | ✅ | 다운된 노드 1개를 담은 리스트 (`[{"name": ..., "fixed_ip": ...}]`) |

**동작 순서**

1. `recovery-provision`이 먼저 실행되어 해당 노드를 재생성합니다.
2. `find-leader` role이 실행되어 현재 생존 중인 리더(`swarm_leader_hostname`, `swarm_leader_ip`)를 찾아둡니다.
3. 생존 리더에서 `docker swarm join-token -q manager`로 매니저 토큰을 발급받습니다.
4. 재생성된 노드에서
   - SSH 연결 대기 → docker 기동 확인
   - 혹시 있을 수 있는 `/var/lib/docker/swarm` 제거
   - docker 재시작으로 메모리상 swarm 상태 초기화
   - 생존 리더에게 `docker swarm join`으로 재조인
5. 재조인한 노드의 새 Docker Node ID를 기록해 이후 유령 노드 정리 단계에서 실수로 삭제되지 않도록 보호합니다.

<br>

### Recovery2: 2대 다운 (쿼럼 상실)

**전제**: 3대 중 2대가 다운됨. 1대만 생존해 있어 raft 쿼럼(과반)이 깨진 상태입니다.

**필수 변수**

| 변수 | 필수 | 설명 |
| --- | :---: | --- |
| `down_nodes` | ✅ | 다운된 노드 정확히 2개를 담은 리스트 |

> Alertmanager의 `group_by`에 `node`가 포함되어 있으면, 2대가 동시에 죽어도 알림이
> 노드별로 개별 그룹으로 쪼개져서 웹훅에 **한 번에 1개씩** 들어올 수 있습니다.
> 이 경우 `down_nodes` 리스트가 항상 1개짜리로만 와서 recovery2의 길이 검증(`==2`)에
> 걸려 실패합니다. `group_by`에서 `node`를 빼서, 같은 alertname의 알림들이 하나로
> 묶여 `alerts[]` 배열에 2개가 한 번에 담기도록 해야 합니다.

**동작 순서**

1. `recovery-provision`이 먼저 실행되어 두 노드를 각각 재생성합니다.
2. 입력값(`down_nodes`)이 정확히 2개인지, 모두 `swarm_managers` 그룹 소속인지 검증합니다.
3. `swarm_managers`에서 다운 목록을 제외한 나머지 1대를 **생존 노드**로 자동 지정합니다.
4. 생존 노드에서 `docker node ls`가 정상 응답하면(쿼럼이 실제로는 살아있으면) 안전을 위해 즉시 중단합니다.
5. 생존 노드에서 자신의 살아있는 raft 데이터를 기반으로
   ```bash
   docker swarm init --force-new-cluster --advertise-addr <생존 노드 IP>
   ```
   를 실행해 단독으로 새 클러스터를 재구성합니다.
6. 매니저 join 토큰을 발급해 두 재생성 노드를 순서대로 재조인시킵니다.

<br>

### Recovery3: 3대(전체) 다운 - S3 백업 복원

**전제**: 매니저 3대가 전부 유실되어, 살아있는 raft 상태 자체가 존재하지 않습니다.
→ 사전에 `swarm/backup` role로 S3에 올려둔 스냅샷으로부터 복원해야 합니다.

**필수 변수**: 없음 (S3에서 최신 백업을 자동으로 찾습니다)

**동작 순서**

1. S3 버킷의 `docker-swarm-backups/` 경로에서 `LastModified` 기준 최신 백업 키를 조회합니다.
2. 백업이 하나도 없으면 즉시 실패 처리합니다 (`swarm_backup` 선행 필요 안내).
   실제로 백업이 없을 경우 AWS CLI가 `sort_by()` 함수 자체에서
   `invalid type for value: None, expected ... 'array'` 에러를 내며 실패합니다.
3. 백업 파일명에서 복원 대상 호스트명을 역산합니다.
4. mgmt(로컬 컨트롤 노드)로 백업 파일을 내려받아 무결성을 먼저 검증합니다.
5. 복원 대상 노드에서
   - SSH 연결 대기
   - docker 정지 → `dockerd` 프로세스가 완전히 종료될 때까지 대기
   - 기존 `/var/lib/docker/swarm` 삭제
   - 백업 파일을 노드로 전송 후 권한까지 그대로 복원
   - 복원된 `swarm` 디렉터리 소유권/권한을 `root:root 0700`으로 재설정
   - docker 기동 → `docker swarm init --force-new-cluster`로 복원된 raft 상태 기반 재구성
6. 나머지 2대는 Recovery1/2와 동일하게 상태 초기화 후 새 리더에 재조인합니다.

<br>

## 공통 후처리: 유령 노드 정리 & 라벨링

세 시나리오 모두 재조인이 끝난 뒤 동일한 마무리 작업을 수행합니다.

- **유령(중복) 노드 정리**: 재생성된 노드가 예전과 같은 호스트명으로 다시 조인하면,
  `docker node ls`에 같은 호스트명이 두 번 나타날 수 있습니다. `Ready` 상태가 아닌 쪽만 골라 제거합니다.
- **라벨 재적용**: 현재 리더 노드에는 `role=leader`, 나머지 모든 노드에는 `role=was` 라벨을 다시 붙입니다.
- **최종 검증**: `docker node ls` 결과를 출력해 복구가 정상적으로 끝났는지 확인할 수 있게 합니다.

<br>

## webhook을 통한 자동 트리거

Prometheus 알림 → Alertmanager → `ansible-webhook.py`가 다음과 같이 자동 연동됩니다.

1. Prometheus rules가 매니저 노드별 `up == 0` 상태를 감지하면서, 동시에 "지금 몇 대가
   생존해 있는지"(`swarm:manager_active_count`)에 따라 `SwarmManagerOneDown` /
   `SwarmManagerTwoDown` 알림을 발생시킵니다. 이때 `and on()` 벡터 매칭으로,
   알림에 실제로 죽은 노드의 `node`/`instance` 라벨이 그대로 실립니다.
2. Alertmanager는 `action` 라벨(`recovery_one`/`recovery_two`)에 따라
   `http://<mgmt-ip>:8080/recovery/one` 또는 `/recovery/two`로 webhook을 보냅니다.
3. `ansible-webhook.py`는 payload의 `alerts[].labels.node`를 읽어 `down_nodes` 리스트를
   동적으로 구성하고, 해당하는 `swarm_recoveryN` 태그로 ansible-playbook을 실행합니다.
4. payload에서 노드 정보를 하나도 못 찾으면, 안전을 위해 사전 정의된 폴백값으로 대체 실행됩니다.

<br>

## 실행 방법

```bash
# 1대 다운 복구 (수동 실행 예시)
ansible-playbook site.yml --tags swarm_recovery1 \
  -e '{"down_nodes": [{"name": "swarm-mg2", "fixed_ip": "192.168.1.21"}]}'

# 2대 다운 복구 (수동 실행 예시, 반드시 2개)
ansible-playbook site.yml --tags swarm_recovery2 \
  -e '{"down_nodes": [{"name": "swarm-mg2", "fixed_ip": "192.168.1.21"}, {"name": "swarm-mg3", "fixed_ip": "192.168.1.22"}]}'

# 3대(전체) 다운 복구 (S3 최신 백업에서 자동 복원)
ansible-playbook site.yml --tags swarm_recovery3
```

일반적으로는 위 명령을 사람이 직접 칠 필요 없이, Alertmanager → webhook을 통해 자동 실행됩니다.

<br>

## 주의사항

- 각 재생성 노드에 대해 `wait_for_connection` 타임아웃이 300초로 설정되어 있습니다.
  클라우드 인스턴스 재생성/부팅이 이보다 오래 걸리는 환경이라면 타임아웃 값을 늘려야 합니다.
- webhook에는 동시 실행 방지 락이 걸려 있어, 이미 recovery 플레이북이 실행 중일 때
  들어온 다른 recovery 요청은 무시(스킵)됩니다. 여러 시나리오가 겹쳐 발생하면
  나중에 온 요청이 유실될 수 있으니, Alertmanager의 `group_by` 설정으로 같은 시나리오의
  알림들이 하나로 묶여 오도록 관리하는 것이 중요합니다.
- `recovery-provision` 단계가 인스턴스를 실제로 삭제/재생성하므로, 일시적인 네트워크
  단절로 인한 오탐일 경우에도 인스턴스가 통째로 재생성됩니다. 알림이 실제 장애인지
  모니터링을 통해 먼저 확인하는 것이 안전합니다.
