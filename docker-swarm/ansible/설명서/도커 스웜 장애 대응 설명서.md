## 소개

3대의 매니저(`swarm-mg`, `swarm-mg2`, `swarm-mg3`)로 구성된 Docker Swarm 클러스터에서,
노드가 다운(재생성)되었을 때 몇 대가 죽었는지에 따라 복구 절차와 위험도가 완전히 달라집니다.

- 1대만 죽으면 → 나머지 2대로 쿼럼(quorum)이 유지되므로 **단순 재조인**만으로 복구됩니다.
- 2대가 죽으면 → 쿼럼이 깨지므로, 생존한 1대가 **자신의 raft 상태로 강제 재구성**해야 합니다.
- 3대 전부 죽으면 → 살아있는 raft 상태 자체가 없으므로, **S3에 백업해둔 raft 데이터를 복원**한 뒤 재구성해야 합니다.

이 세 가지를 각각 `swarm/recovery1`, `swarm/recovery2`, `swarm/recovery3` role로 분리해 두었습니다.

<br>

## 복구 시나리오 요약

| 시나리오 | 다운된 대수 | 쿼럼 상태 | 필요 입력 변수 | 핵심 동작 |
| --- | :---: | --- | --- | --- |
| **Recovery1** | 1 / 3 | 유지됨 (2대 생존) | `down_node_hostname` | 생존 리더(`find-leader`가 찾음)의 join 토큰으로 재생성 노드를 재조인 |
| **Recovery2** | 2 / 3 | 상실됨 (1대 생존) | `down_node_hostnames` (콤마 구분, 2개) | 생존 노드가 `--force-new-cluster`로 자신의 raft 기반 재구성 후, 2대 재조인 |
| **Recovery3** | 3 / 3 (전멸) | 없음 (라이브 상태 없음) | (없음, S3 최신 백업 자동 탐색) `vault_s3_bucket` 필요 | S3 최신 백업 다운로드 → 대상 노드에 swarm 디렉터리 복원 → `--force-new-cluster` → 나머지 2대 재조인 |

<br>

## 전체 실행 흐름

전체 순서는 `site.yml`을 그대로 따릅니다. 복구 관련 플레이만 발췌하면 아래와 같습니다.

```yaml
- name: Install Docker
  hosts: swarm_managers
  tags: [docker, swarm_setup, swarm_recovery3, swarm_recovery2, swarm_recovery1]
  roles: [swarm/docker]

- name: Discover current swarm leader
  hosts: swarm_managers
  tags: [swarm_recovery1]
  roles: [swarm/find-leader]

- name: Recovery swarm when 1 node down
  tags: [swarm_recovery1]
  roles: [swarm/recovery1]

- name: Recovery swarm when 2 nodes down
  tags: [swarm_recovery2]
  roles: [swarm/recovery2]

- name: Recovery swarm when 3 nodes down
  tags: [swarm_recovery3]
  roles: [swarm/recovery3]
```

> ⚠️ **Recovery1만 `find-leader`(swarm_leader) 태그를 함께 거칩니다.**
> Recovery2/3는 쿼럼 자체가 깨진 상황을 가정하므로, `docker node ls`로 리더를 조회하는
> `find-leader` 방식이 통하지 않습니다. 대신 role 내부에서 "다운되지 않은 노드"를
> 직접 생존 노드로 지정해 처리합니다.

<br>

## 사전 조건

- 다운된 노드는 **동일한 인벤토리 호스트명으로 재생성(reprovision)** 되어 있음을 가정합니다.
- Recovery3를 쓰려면 사전에 `swarm_backup` 태그로 S3에 백업이 최소 1회 이상 생성되어 있음을 가정합니다.
- S3 접근을 위한 AWS 프로필(`aws_profile_name`, 기본값 `swarm-backup`)과 `vault_s3_bucket` 변수(버킷명)가 vault 등에 정의해두었습니다.

<br>

## 시나리오별 상세

### Recovery1: 1대 다운 (쿼럼 유지)

**전제**: 3대 중 1대만 재생성됨. 나머지 2대는 정상 동작 중이라 쿼럼이 살아있습니다.

**필수 변수**

| 변수 | 필수 | 설명 |
| --- | :---: | --- |
| `down_node_hostname` | ✅ | 재생성된 노드의 인벤토리 호스트명 (예: `swarm-mg2`) |

**동작 순서**

1. `find-leader` role이 먼저 실행되어 현재 생존 중인 리더(`swarm_leader_hostname`, `swarm_leader_ip`)를 찾아둡니다.
2. 생존 리더에서 `docker swarm join-token -q manager`로 매니저 토큰을 발급받습니다.
3. 재생성된 노드(`down_node_hostname`)에서
   - SSH 연결 대기 → docker 기동 확인
   - 혹시 있을 수 있는 `/var/lib/docker/swarm` 제거
   - docker 재시작으로 메모리상 swarm 상태 초기화
   - 생존 리더에게 `docker swarm join`으로 재조인
4. 재조인한 노드의 새 Docker Node ID를 기록해 이후 유령 노드 정리 단계에서 실수로 삭제되지 않도록 보호합니다.

<br>

### Recovery2: 2대 다운 (쿼럼 상실)

**전제**: 3대 중 2대가 재생성됨. 1대만 생존해 있어 raft 쿼럼(과반)이 깨진 상태입니다.

**필수 변수**

| 변수 | 필수 | 설명 |
| --- | :---: | --- |
| `down_node_hostnames` | ✅ | 다운된 두 노드의 인벤토리 호스트명, 콤마로 구분 (예: `swarm-mg2,swarm-mg3`) |

**동작 순서**

1. 입력값을 콤마 기준으로 파싱해 2개인지, 모두 `swarm_managers` 그룹 소속인지 검증합니다.
2. `swarm_managers`에서 다운 목록을 제외한 나머지 1대를 **생존 노드**로 자동 지정합니다.
3. 그럴 일은 없겠지만, 생존 노드에서 `docker node ls`가 정상 응답하면(쿼럼이 실제로는 살아있으면) **안전을 위해 즉시 중단**합니다.
4. 생존 노드에서 자신의 살아있는 raft 데이터를 기반으로
   ```bash
   docker swarm init --force-new-cluster --advertise-addr <생존 노드 IP>
   ```
   를 실행해 단독으로 새 클러스터를 재구성합니다.
5. 매니저 join 토큰을 발급해 두 재생성 노드를 순서대로 재조인시킵니다.

<br>

### Recovery3: 3대(전체) 다운 - S3 백업 복원

**전제**: 매니저 3대가 전부 유실/재생성되어, 살아있는 raft 상태 자체가 존재하지 않습니다.
→ 사전에 `swarm/backup` role로 S3에 올려둔 스냅샷으로부터 복원해야 합니다.

**필수 변수**: 없음 (S3에서 최신 백업을 자동으로 찾습니다)

**동작 순서**

1. S3 버킷(`vault_s3_bucket`)의 `docker-swarm-backups/` 경로에서 `LastModified` 기준 최신 백업 키를 조회합니다.
2. 백업이 하나도 없으면 즉시 실패 처리합니다 (`swarm_backup` 선행 필요 안내).
3. 백업 파일명(`<호스트명>-swarm-backup-<타임스탬프>.tar.gz`)에서 **복원 대상 호스트명을 역산**합니다.
4. mgmt(로컬 컨트롤 노드)로 백업 파일을 내려받아 `tar -tzvf`로 무결성을 먼저 검증합니다.
5. 복원 대상 노드에서
   - SSH 연결 대기
   - docker 정지 → `dockerd` 프로세스가 완전히 종료될 때까지 대기(최대 30초)
   - 기존 `/var/lib/docker/swarm` 삭제
   - 백업 파일을 노드로 전송 후 `tar -xzvpf`로 **권한까지 그대로 복원**
   - 복원된 `swarm` 디렉터리 소유권/권한을 `root:root 0700`으로 재설정 (틀리면 dockerd가 즉시 크래시)
   - docker 기동 → 로그에서 raft panic 여부 확인(참고용 로그만 남기고, 이후 처리는 동일)
   - `docker swarm init --force-new-cluster`로 복원된 raft 상태 기반 재구성
6. 나머지 2대(복원 대상이 아닌 노드)는 Recovery1/2와 동일하게 상태 초기화 후 새 리더에 재조인합니다.

## 공통 후처리: 유령 노드 정리 & 라벨링

세 시나리오 모두 재조인이 끝난 뒤 동일한 마무리 작업을 수행합니다.

- **유령(중복) 노드 정리**: 재생성된 노드가 예전과 같은 호스트명으로 다시 조인하면,
  `docker node ls`에 같은 호스트명이 두 번 나타날 수 있습니다.
  `Hostname` 기준으로 중복을 찾아, `Ready` 상태가 아닌 쪽(예전 유령 항목)만 골라 제거합니다.
  이때 매니저 role이면 raft 안전성 때문에 강제 삭제가 거부되므로, 먼저 `docker node demote` 후 삭제합니다.
  방금 재조인한 새 노드 ID는 항상 삭제 대상에서 제외됩니다.
- **라벨 재적용**: 현재 리더 노드에는 `role=leader`, 나머지 모든 노드에는 `role=was` 라벨을 다시 붙여
  이전 배포(`swarm/labels`, `swarm/deploy`) 단계와의 정합성을 맞춥니다.
- **최종 검증**: `docker node ls` 결과를 출력해 복구가 정상적으로 끝났는지 눈으로 확인할 수 있게 합니다.

<br>

## 실행 방법

```bash
# 1대 다운 복구
ansible-playbook site.yml --tags swarm_recovery1 \
  -e down_node_hostname=swarm-mg2

# 2대 다운 복구
ansible-playbook site.yml --tags swarm_recovery2 \
  -e "down_node_hostnames=swarm-mg2,swarm-mg3"

# 3대(전체) 다운 복구 (S3 최신 백업에서 자동 복원)
ansible-playbook site.yml --tags swarm_recovery3
```

> 실행 전 인벤토리에서 다운된 노드가 재생성된 새 인스턴스를 가리키고 있는지 반드시 확인하세요.
> (IP가 바뀌었다면 인벤토리 갱신이 선행되어야 합니다.)

<br>

## 주의사항

- 각 재생성 노드에 대해 `wait_for_connection` 타임아웃이 300초로 설정되어 있습니다.
  클라우드 인스턴스 재생성/부팅이 이보다 오래 걸리는 환경이라면 `restore_wait_for_connection_timeout`
  값을 늘려야 합니다.
