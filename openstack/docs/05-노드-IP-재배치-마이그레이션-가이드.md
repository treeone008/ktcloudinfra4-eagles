# 노드 IP 재배치 마이그레이션 가이드 (2026-06-29)

> **목표:** 합의된 IP표에 맞춰 노드 관리 IP 변경. **운영 중 클러스터라 신중히.**
> control(.100)/compute2(.104)/VIP(.105)는 그대로 → RabbitMQ·MariaDB가 control에 있어 위험 낮음.

## 변경표 (회전 — 순서 지키면 충돌 없음)

| 노드 | 현재 | 목표 | 비고 |
|------|------|------|------|
| compute1 | 172.16.8.101 | **172.16.8.103** | ① 먼저 (storage off라 .103 비어있음) |
| network | 172.16.8.102 | **172.16.8.101** | ② (compute1 빠진 자리) |
| storage | 172.16.8.103 | **172.16.8.102** | ③ storage 켤 때 (지금 OFF, 오늘은 보류) |
| control | 172.16.8.100 | 172.16.8.100 | 변경 없음 |
| compute2 | 172.16.8.104 | 172.16.8.104 | 변경 없음 |
| VIP/Horizon | 172.16.8.105 | 172.16.8.105 | 변경 없음 |

> 오늘은 **compute1 + network**만. storage(.102)는 다시 켤 때 적용.

---

## ⚠️ 핵심 위험 & 대비

- netplan은 **각 노드 VMware 콘솔에서** 변경(SSH로 하면 끊김).
- 모든 노드 `/etc/hosts` + control `/etc/kolla/multinode` 인벤토리 동기화 필수.
- 변경 후 **`kolla-ansible reconfigure`**로 렌더링 config(my_ip, neutron 터널 local_ip 등) 갱신해야 정상화.
- neutron OVS 터널(local_ip)이 노드 IP 기반이라, reconfigure 전까지 db 인스턴스 통신이 끊길 수 있음(정상).

## 0. 사전 백업 (control)
```bash
sudo -i
cp /etc/kolla/multinode /etc/kolla/multinode.bak-0629
cp /etc/hosts /etc/hosts.bak-0629
```
각 노드에서도:
```bash
cp /etc/netplan/50-cloud-init.yaml /root/50-cloud-init.yaml.bak-0629
cp /etc/hosts /root/hosts.bak-0629
```

---

## STEP 1. compute1 : .101 → .103 (VMware 콘솔)

compute1 콘솔(root):
```bash
sed -i 's/172\.16\.8\.101/172.16.8.103/' /etc/netplan/50-cloud-init.yaml
cat /etc/netplan/50-cloud-init.yaml      # .103 확인
netplan apply
ip -4 a | grep 172.16.8.103
```

## STEP 2. network : .102 → .101 (VMware 콘솔)

network 콘솔(root):
```bash
sed -i 's/172\.16\.8\.102/172.16.8.101/' /etc/netplan/50-cloud-init.yaml
cat /etc/netplan/50-cloud-init.yaml
netplan apply
ip -4 a | grep 172.16.8.101
```

## STEP 3. 전 노드 /etc/hosts 동기화

control·compute1·network·compute2 각각 `/etc/hosts`가 아래가 되도록:
```
172.16.8.100 control
172.16.8.103 compute1
172.16.8.101 network
172.16.8.102 storage
172.16.8.104 compute-node-02
```
> compute1=103, network=101, storage=102(예정)로 정정.

## STEP 4. control: 인벤토리 갱신

`/etc/kolla/multinode`:
- compute1: `172.16.8.101` → `172.16.8.103`
- network 그룹의 `172.16.8.102` → `172.16.8.101`
- storage 주석도 `#172.16.8.103` → `#172.16.8.102` (혼동 방지)

```bash
sed -i 's/^172\.16\.8\.101/172.16.8.103/' /etc/kolla/multinode     # compute1
sed -i 's/^172\.16\.8\.102/172.16.8.101/' /etc/kolla/multinode     # network
sed -i 's/^#172\.16\.8\.103/#172.16.8.102/' /etc/kolla/multinode   # storage(주석)
grep -nE '172.16.8.10[1-4]' /etc/kolla/multinode
```

## STEP 5. SSH known_hosts 정리 + 연결 확인 (control)
```bash
ssh-keygen -f ~/.ssh/known_hosts -R 172.16.8.101   # 옛 키 제거
ssh-keygen -f ~/.ssh/known_hosts -R 172.16.8.103
ssh root@172.16.8.103 hostname    # compute1
ssh root@172.16.8.101 hostname    # network

source /root/venv/bin/activate
ansible -i /etc/kolla/multinode all -m ping        # 전 노드 SUCCESS
```

## STEP 6. kolla 재구성 (config 갱신)
```bash
source /root/venv/bin/activate
kolla-ansible reconfigure -i /etc/kolla/multinode
```
> 전체 reconfigure (storage 주석 상태). 5~15분.

## STEP 7. 검증
```bash
source /etc/kolla/admin-openrc.sh
openstack compute service list        # compute1·compute-node-02 up
openstack network agent list          # network·compute agents up
openstack server list                 # db01·db02 ACTIVE

# private ping (network 노드에서 namespace)
ssh root@172.16.8.101 "ip netns list"
ssh root@172.16.8.101 "ip netns exec qdhcp-3b1d5e65-8c25-49ca-9dff-c647ee671c4c ping -c3 192.168.101.31"
```

## ★ 실전 함정 (2026-06-29 실제 발생)

**증상:** IP 재배치 + reconfigure 후, compute1의 db는 핑 되는데 **compute-node-02의 db만 핑 불통**.

**원인:** reconfigure RECAP에서 **`compute-node-02 ... changed=0`** — 즉 IP가 안 바뀐 compute-node-02는 아무 변경도 안 됐다. 그런데 **peer인 network 노드 IP가 .102→.101로 바뀌어서**, compute-node-02의 OVS 에이전트가 옛 터널 엔드포인트를 물고 있어 VXLAN 터널이 끊김.

**해결:** IP 안 바뀐 노드라도 **OVS 에이전트 재시작**으로 터널 갱신:
```bash
ssh root@172.16.8.104 "docker restart neutron_openvswitch_agent"
sleep 30   # 후 핑 재시도
```
> 교훈: 노드 IP 마이그레이션 시 **IP가 안 바뀐 compute 노드들도** OVS 에이전트를 한 번씩 재시작(또는 `--limit`로 강제 reconfigure)해서 터널을 새 peer IP로 다시 맺어줘야 한다.


## storage (나중에 켤 때)
1. storage 콘솔: `sed -i 's/172\.16\.8\.103/172.16.8.102/' /etc/netplan/50-cloud-init.yaml && netplan apply`
2. control 인벤토리: `#172.16.8.102` → 주석 해제(`172.16.8.102`)
3. `kolla-ansible bootstrap-servers/deploy --limit 172.16.8.102` 또는 reconfigure
