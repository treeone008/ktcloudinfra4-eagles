# 07 — 시연 PC 전달(스냅샷/패키징) & 복원 가이드

> **목적:** 내(김현도) PC에서 만든 OpenStack 환경을 **팀장 시연 PC로 전달 → 그대로 다시 구동**할 때
> 헷갈리지 않도록 정리한 가이드. (패키징 방법 + 복원 순서 + 계정 정보 + 함정 체크리스트)
>
> 작성: 김현도 | 환경: VMware Workstation 멀티노드 / Ubuntu 24.04 / Kolla-Ansible

|------|-----|-----|

## 1. 전달 단위 — "노드 세트 통째로"

OpenStack 멀티노드는 **여러 VM이 한 클러스터**라서, 일부만 주면 안 되고 **세트로** 줘야 함.

| 묶음 | 포함 VM | 비고 |
|------|---------|------|
| **최소 동작 세트** | control + network + storage + compute1 | tenant VM 띄우려면 이 4개 필수 |
| **권장 세트** | + compute-node-02 | db anti-affinity 분산까지 시연 |
| (선택) | + mgmt-01 | 관리/SSH 진입점 |

## 2. 패키징 방법 (2가지 중 택1)

> ⚠️ **스냅샷(Snapshot)은 "되돌리기용"이라 단독 전달 불가.** PC 간 이동은 아래 둘 중 하나.

### 방법 A — VM 폴더 통째 압축 (제일 쉬움, 추천)
1. VMware에서 **모든 노드 완전 종료**(Shut Down Guest, 일시정지 X).
2. 각 VM 폴더(`*.vmx`, `*.vmdk` 들어있는 폴더)를 통째로 zip/7z 압축.
3. 노드별로 압축하거나, 상위 폴더를 한 번에 압축.
4. 시연 PC로 전송(외장하드/네트워크). **용량 크니 미리 확인.**

### 방법 B — OVF/OVA 내보내기 (표준 포맷)
1. VMware: `File → Export to OVF...` (VM별로)
2. 생성된 `.ovf` + `.vmdk`(또는 `.ova`) 전달.
3. 시연 PC: `File → Open...` 으로 `.ovf/.ova` 열어서 임포트.

> 둘 다 결과는 같음. A가 빠르고, B가 깔끔(표준). 우리 팀 작업방식 "만들기→압축→전송"엔 **A**가 잘 맞음.

## 3. 시연 PC 사전 준비 (복원 전)

### 3-1. VMware 네트워크(NAT) 대역 맞추기 ★중요
우리 환경은 `172.16.8.0/24` (NAT)예요. 시연 PC도 **같은 대역**이어야 노드끼리 통신됨.

1. VMware `Edit → Virtual Network Editor` (관리자 권한)
2. **vmnet8 (NAT)** 선택
3. Subnet IP: `172.16.8.0`, Subnet mask: `255.255.255.0`
4. `NAT Settings...` → Gateway IP: `172.16.8.2`
5. 적용 후 저장

> 이게 안 맞으면 노드는 켜져도 서로 못 찾아서 OpenStack 전체가 죽은 것처럼 보임.

### 3-2. 호스트 자원 확인
- RAM: 최소 세트 4노드 합산 RAM 이상 (control 6 + network 2 + storage 3 + compute 6 ≈ 17GB+). 여유 있게 24GB↑ 권장.
- 디스크: 압축 푼 용량 + 여유.

## 4. 복원 순서 (시연 PC) ★순서 지키기

> nova/neutron이 control·network에 의존하므로 **순서대로** 켜야 깔끔하게 올라옴.

1. **control** 부팅 → 로그인(`user1`/`user1`) → 완전히 올라올 때까지 대기 (1~2분)
2. **network** 부팅
3. **storage** 부팅
4. **compute1** 부팅
5. **compute-node-02** 부팅
6. (있으면) **mgmt-01** 부팅

> 부팅 시 VMware가 "이동했습니까? 복사했습니까?" 물으면 → **"I Moved It"(이동함) 선택.**
> ("I Copied It"을 고르면 MAC/UUID가 바뀌어 네트워크/식별자 꼬일 수 있음.)

## 5. 복원 후 검증 (control에서)

```bash
sudo -i                                   # 비번 test123 또는 user1
source /root/venv/bin/activate
source /etc/kolla/admin-openrc.sh

# 1) 노드/서비스 살아있나
openstack compute service list            # nova-compute (compute1, compute-node-02) up
openstack network agent list              # 전부 Alive :-)   ← ACTIVE 핵심
openstack volume service list             # cinder up

# 2) 리소스/인스턴스
openstack server list                     # swarm-mg-01, db01, db02 ACTIVE
openstack server show db01 -c status -c "OS-EXT-SRV-ATTR:host" -c addresses
```

### 잘 안 올라올 때
| 증상 | 조치 |
|------|------|
| agent가 `XXX`(죽음) | 해당 노드: `docker restart neutron_openvswitch_agent neutron_l3_agent neutron_dhcp_agent` |
| compute의 인스턴스만 통신 불가 | 그 compute: `docker restart neutron_openvswitch_agent` (터널 갱신) |
| nova-compute down | compute 노드: `docker restart nova_compute` |
| 라우터 DOWN | `openstack network agent list | grep l3` 확인 후 `neutron_l3_agent` 재시작 |
| 노드끼리 핑 불가 | §3-1 vmnet8 대역(172.16.8.0/24) 재확인 |

> 자세한 네트워크/라우터 트러블슈팅: `06-네트워크-라우터-구성-및-ACTIVE-트러블슈팅-가이드.md`

## 6. ⚠️ 절대 하지 말 것 (우리가 실제로 당한 함정)

| 금지 | 이유 |
|------|------|
| **원본 PC와 시연 PC에서 같은 노드 동시 구동** | 같은 host명/nova host가 **같은 RabbitMQ 응답큐를 공유** → 양쪽 compute가 `MessagingTimeout`으로 죽음 (금~월 삽질의 진범). 시연 시 **내 원본은 꺼둘 것.** |
| compute 노드를 클론해서 추가 | `host=compute1`이 복제돼 응답큐 충돌 → 신규 설치가 정답 (`04` 가이드) |
| 부팅 시 "I Copied It" 선택 | MAC/UUID 변경 → 네트워크·식별자 꼬임. **"I Moved It"** 선택 |
| 노드 일부만 전달 | 클러스터 성립 안 됨 → 최소 세트(§1) 통째로 |

- [ ] **이 문서 + `오픈스택-구성-문서화.md`** 같이 전달 (IP·계정·구성 한눈에)

- [ ] 시연 PC vmnet8 = `172.16.8.0/24`, GW `172.16.8.2` 안내

## 8. 한 줄 요약

> **전달 = 노드 세트를 "완전 종료 → 폴더 압축(or OVF)" 해서 통째로.**
> **시연 PC = vmnet8을 172.16.8.0/24로 맞추고 → control부터 순서대로 부팅("I Moved It") → `network agent list`로 확인.**
> **계정은 전부 `user1`(안 되면 test123), 내 원본과 동시 구동 절대 금지.**
