### 🚧 진행 중 겪은 문제점 — 그리고 해결

**★ 근본원인: 클론 compute의 `host` 충돌 (금~월 삽질의 진범)**

- compute1을 **Full Clone**하면 nova `[DEFAULT] host = compute1`이 그대로 복사됨.
- 클론(compute2)이 자기를 compute1로 착각 → RabbitMQ 응답큐 `reply_compute1:nova-compute`를 **공유**(컨슈머 2개).
- conductor 응답이 둘에게 번갈아 배달 → 절반이 엉뚱한 쪽으로 → 양쪽 `MessagingTimeout`.
- 증상: `amqpdriver No calling threads waiting for msg_id` + `MessagingTimeout`(`init_host → _get_nodes → conductor.call`).
- **compute1까지 끌려 내려가 down**이 됨.

**해결 과정:**
1. 진단: RabbitMQ `list_queues`에서 `reply_compute1:nova-compute` **컨슈머 2개** 발견 → 경쟁 컨슈머 확정.
2. compute2 nova_compute 정지 → 자동 재시작(`restart=always`)으로 계속 부활 → **compute2 VM 전원 OFF**로 차단 → **compute1 정상 회복**(`up`).
3. 클론은 구조적 한계 → **깨끗한 새 VM으로 compute2 재제작** → `host=compute-node-02`로 분리.
4. 검증: 응답큐가 `reply_compute1` / `reply_compute-node-02`로 **분리**, 각 컨슈머 1 → 재발 불가 확인.

> 교훈: **compute 노드는 클론하지 말 것.** 클론하면 host/compute_id/응답큐가 겹쳐서 기존 compute까지 죽인다. 신규 설치가 정답.
