### 🚧 진행 중 겪은 문제점 — 그리고 해결

1. **재부팅 후 인스턴스 자동 시작 안 됨** (전부 SHUTOFF)
   - → `openstack server start`로 수동 기동. (추후 `resume_guests_state_on_host_boot=true`로 자동화 가능)

2. **재부팅 후 FIP 핑 단절** (control조차 .219 불통)
   - → network 노드 L3/OVS 에이전트 재시작으로 qrouter/FIP 경로 복구. (어제 IP재배치 때 OVS 재시작과 동일 류)

3. **mgmt가 FIP(.219)에 직접 SSH 불가** (`Destination Host Unreachable` = ARP 실패)
   - 노드는 NIC 2개(ens33 관리망 + ens34 provider/br-ex)인데 **mgmt는 NIC 1개**라 provider 세그먼트에 발이 없음.
   - 2번째 NAT NIC(ens37) 추가 + 호스트 라우트로 테스트했으나 **여전히 불통** → VMware NAT + neutron L2가 얽힌 깊은 이슈로 판단.
   - **→ 우회 채택: control 경유 ProxyJump.** 실무 bastion의 정상 패턴이며 "mgmt SSH 가능" 산출물 충족. (직접연결은 추후 별도 진단)

4. **ProxyJump가 자꾸 "멈춤"**
   - 원인: control 점프 단계에서 **비밀번호(test123) 입력 대기** (MobaXterm에서 프롬프트 안 보임).
   - → mgmt 공개키를 control authorized_keys에 등록(`ssh-copy-id`)해 **무비번 점프**로 정리 중.

5. **mgmt 세션 잦은 끊김** (MobaXterm) — 호스트 RAM 압박 추정 (VM 6대 동시 ~23GB).
   - → 필요 시 storage 잠시 종료로 RAM 확보 가능. 작업은 control 직접 세션 병행으로 회피.
