## 1. 노드 IP 재배치 (회전)

> 회전이라 **순서**가 중요(충돌 방지): compute1(.101→.103) → network(.102→.101) → storage(.103→.102).

### 1-1. 백업 (control + 각 노드)
```bash
# control
cp /etc/kolla/multinode /etc/kolla/multinode.bak-0629
cp /etc/hosts /etc/hosts.bak-0629
# 각 노드
cp /etc/netplan/50-cloud-init.yaml /root/50-netplan.bak-0629
```

### 1-2. netplan 변경 (각 노드 VMware 콘솔에서)
```bash
# compute1 콘솔
sed -i 's/172\.16\.8\.101/172.16.8.103/' /etc/netplan/50-cloud-init.yaml && netplan apply
# network 콘솔
sed -i 's/172\.16\.8\.102/172.16.8.101/' /etc/netplan/50-cloud-init.yaml && netplan apply
# storage 콘솔 (네트워크 어댑터 분리 후 켜서 변경 → 재연결, .103 충돌 방지)
sed -i 's/172\.16\.8\.103/172.16.8.102/' /etc/netplan/50-cloud-init.yaml && netplan apply
```

### 1-3. 전 노드 /etc/hosts 동기화 (control·compute1·network·compute2 각각)
```bash
sed -i '/172\.16\.8\.10[1-4]/d' /etc/hosts
cat >> /etc/hosts <<'EOF'
172.16.8.103 compute1
172.16.8.101 network
172.16.8.102 storage
172.16.8.104 compute-node-02
EOF
```

### 1-4. control 인벤토리 갱신 (placeholder 방식 — 체인 치환 버그 방지)
```bash
sed -i -E \
  -e 's/^172\.16\.8\.101\s*$/__C1__/' \
  -e 's/^172\.16\.8\.102\s*$/__NET__/' \
  -e 's/^#?172\.16\.8\.103\s*$/__STO__/' \
  -e 's/^__C1__$/172.16.8.103/' \
  -e 's/^__NET__$/172.16.8.101/' \
  -e 's/^__STO__$/172.16.8.102/' \
  /etc/kolla/multinode
```

### 1-5. SSH known_hosts 정리 + ansible 확인
```bash
for ip in 172.16.8.101 172.16.8.102 172.16.8.103; do ssh-keygen -f /root/.ssh/known_hosts -R $ip; done
ssh -o StrictHostKeyChecking=accept-new root@172.16.8.101 hostname   # network
ssh -o StrictHostKeyChecking=accept-new root@172.16.8.102 hostname   # storage
ssh -o StrictHostKeyChecking=accept-new root@172.16.8.103 hostname   # compute1
source /root/venv/bin/activate
ansible -i /etc/kolla/multinode all -m ping       # 전 노드 SUCCESS
```

### 1-6. kolla reconfigure
```bash
kolla-ansible reconfigure -i /etc/kolla/multinode    # 전부 failed=0
```

### ★ 함정 (실제 발생)
- reconfigure RECAP에서 **`compute-node-02 changed=0`** → IP 안 바뀐 노드는 손 안 댐.
- 그런데 peer(network)가 .102→.101로 바뀌어 **compute-node-02의 OVS 터널이 옛 IP를 물고 있어 db01 핑 불통**.
- **해결:** `ssh root@172.16.8.104 "docker restart neutron_openvswitch_agent"` → 터널 갱신 → 핑 정상.
- 교훈: IP 마이그레이션 시 **IP 안 바뀐 compute도 OVS 에이전트 재시작** 필요.

## 2. private(사설망) 인스턴스 핑

> 사설 IP(192.168.x)는 control에 인터페이스가 없어 직접 못 닿음 → **network 노드의 namespace** 통해 핑.

```bash
source /etc/kolla/admin-openrc.sh
PRIV=$(openstack network show project-private-net -f value -c id)   # UUID 자동
ssh root@172.16.8.101 "ip netns exec qdhcp-$PRIV ping -c3 192.168.101.31"   # db01
ssh root@172.16.8.101 "ip netns exec qdhcp-$PRIV ping -c3 192.168.101.32"   # db02

# swarm-mg-01은 FIP 있어 namespace 없이 가능
ping -c3 172.16.8.219
```
- `qdhcp-<네트워크ID>` = 그 네트워크 전용 DHCP namespace (사설망에 발 걸침).
- 결과: db01/db02/swarm-mg-01 모두 `0% packet loss`.

## 3. Ubuntu 24.04(noble) 이미지 등록

```bash
cd /root
wget https://cloud-images.ubuntu.com/noble/current/noble-server-cloudimg-amd64.img
source /etc/kolla/admin-openrc.sh
openstack image create "ubuntu" \
  --file /root/noble-server-cloudimg-amd64.img \
  --disk-format qcow2 --container-format bare --public
openstack image delete ubuntu-22.04     # 22.04 정리(이름 ubuntu로 통일)
```
- 등록된 이미지: `ubuntu` (24.04), id `db848620-7131-4b71-baa4-3c501ffcc833`
- 이후 인스턴스 생성은 `--image ubuntu`.

### 3-1. db01/db02를 24.04로 재생성 (anti-affinity 유지)
```bash
GID=$(openstack server group list -f value -c ID -c Name | awk '/db-server-group/{print $1}')
openstack server delete db01 db02
openstack server create --flavor m1.small --image ubuntu \
  --nic net-id=project-private-net,v4-fixed-ip=192.168.101.31 \
  --security-group project-sg --key-name project_key --hint group=$GID db01
openstack server create --flavor m1.small --image ubuntu \
  --nic net-id=project-private-net,v4-fixed-ip=192.168.101.32 \
  --security-group project-sg --key-name project_key --hint group=$GID db02
```
> ⚠️ 두 개를 동시에 만들면 스케줄 레이스로 한쪽이 `No valid host`(ERROR) 날 수 있음 → **순차 생성** 권장. (실제로 db01 ERROR → 삭제 후 재생성하니 compute-node-02로 정상 배치)

- 결과: **db01=compute-node-02, db02=compute1** (anti-affinity 유지), 둘 다 ACTIVE.

## 5. swarm-mg-01 24.04 재생성 (저녁 추가)

> 삭제된 22.04 이미지 참조 정리 → `ubuntu`(24.04)로 재생성.

```bash
# 캡처: fixed 192.168.100.20 / FIP 172.16.8.219 / project-public-net / swarm-server-group / project-sg / project_key
# 삭제 후 유령 포트(192.168.100.20)가 남아 "Fixed IP already in use" 발생 → 포트 정리 필요
PORT=$(openstack port list --fixed-ip ip-address=192.168.100.20 -f value -c id)
openstack floating ip unset --port 172.16.8.219
openstack port delete $PORT                       # fixed IP 해방

NET=$(openstack network show project-public-net -f value -c id)
openstack server create --flavor m1.small --image ubuntu \
  --nic net-id=$NET,v4-fixed-ip=192.168.100.20 \
  --security-group project-sg --key-name project_key \
  --hint group=61f54db2-0c66-4a43-af91-62bbcfa8ca52 swarm-mg-01

openstack server add floating ip swarm-mg-01 172.16.8.219
ssh-keygen -f /root/.ssh/known_hosts -R 172.16.8.219
ssh -i /root/.ssh/id_rsa ubuntu@172.16.8.219 "hostname; grep VERSION= /etc/os-release"
# → swarm-mg-01 / VERSION="24.04.4 LTS (Noble Numbat)"  ✅
```

- ★ 함정: server delete 후 **named port가 안 지워져** fixed IP가 점유됨 → `port delete`로 해방해야 재생성 가능.
- 결과: swarm-mg-01 ACTIVE@compute1, 24.04.4, FIP·키페어 정상.
